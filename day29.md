#데이터 모델링(엔티티/속성)
## 데이터 모델링이란?
데이터 모델링 = 현실 세계를 DB 구조로 옮기는 설계 작업
> “어떤 데이터를 어떤 테이블에 어떤 관계로 어떻게 저장할 것인가”

## 데이터 모데링의 3단계
### 개념적 모델링
비즈니스 관점/기술 요소 X/엔티티, 관계만 정의

ex)회원 주문 상품

### 논리적 모델링
속성 정의/식별자(PK) 정의/정규화 수행/DBMS 독립적

ex)회원(회원ID, 이름, 이메일)

``주문(주문ID, 회원ID, 주문일)

### 물리적 모델링
실제 DB 구현

인덱스, 파티션, 타입 결정

DBMS 종속

## 엔티티

### 유형 엔티티
사람/사물, 독립적 존재

### 개념 엔티티
물리적 실체X, 개념 관리 목적

### 사건 엔티티
행위/이력, 시간 흐름 포함
“업무에서 필요하고,
저장할 가치가 있으며,
식별이 가능한 객체”

### 엔티티 성립 조건 

	업무에서 사용된다

	식별자(PK)가 있다

	2개 이상의 속성을 가진다

	지속적으로 관리된다	
    
엔티티 예시

| 엔티티  | 가능 여부 | 이유           |
| ---- | ----- | ------------ |
| 회원   | O     | 식별 가능, 관리 대상 |
| 주문   | O     | 트랜잭션 대상      |
| 나이   | X     | 속성이지 엔티티 아님  |
| 오늘날씨 | X     | 지속 관리 X     |

## 속성
엔티티의 세부 정보

ex) 회원: 회원ID/이름/이메일/가입일

하나의 속성은 하나의 의미만

업무적으로 의미 있어야 함

계산 가능하면 파생 속성

반복되면 별도 엔터티 고려


### 기본 속성
업무에서 직접 관리

ex) 이름,가격

### 설계 속성
시스템이서 필요

ex) 등록일, 수정일

### 파생 속성
계산으로 얻음, 파생 속성은 가급적 DB에 저장하지 않는다

ex)총금액



## 식별자
엔티티를 유일하게 구분하는 속성

| 구분       | 설명               |
| -------- | ---------------- |
| 본질 식별자   | 주민번호, 사번         |
| 인조 식별자  | 시스템 생성 ID (회원ID) |
| 단일 식별자   | PK 1개            |
| 복합 식별자   | PK 2개 이상         |

인조 식별자 + 단일 PK 선호

## 관계
엔티티 간의 연관성

## ERD 읽기
실선 관계 → 필수 관계

점선 관계 → 선택 관계

까치발 → N

원(O) → 선택

막대(|) → 필수
